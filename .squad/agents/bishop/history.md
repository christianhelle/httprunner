## Learnings

- Bishop owns `src/core` and `src/cli`, including parser, runner, processor, and CLI behavior.
- The repo's core responsibilities include `.http` parsing, request chaining, assertions, variables, and reporting.
- Initial review scope includes possible defects plus security and performance findings that touch execution flow.
- **2026-03-20 Core/CLI audit completed.** Filed GitHub issues #231-#235 for exit codes, JSON extraction correctness, condition parsing, parser strictness, and fail-open request-variable substitution.
- CLI execution failures do not currently propagate to the process exit code because `run()` ignores `ProcessorResults.success` and only exits non-zero on hard `Err` returns.
- Request-variable extraction and `@if` JSON body conditions both depend on `src/core/src/variables/json.rs`, whose string-scanning approach can select nested duplicate keys and truncate object values when braces appear inside JSON strings.
- Condition parsing treats `==` as literal expected text, so `@if login.response.status == 200` silently becomes a never-matching comparison.
- Parser directives and request-variable substitution currently favor warnings/pass-through over hard failures, which makes malformed `@timeout` directives and unresolved `{{request.response...}}` placeholders execute as if they were valid inputs.
- **2026-03-20 Pest.rs migration investigation completed.** Analyzed handwritten parser contract, identified grammar-level vs. semantic post-processing boundaries, mapped all API consumers (CLI, GUI, TUI, serializer), documented critical quirks (directive precedence, state machine, body mode), and produced comprehensive 5-phase migration plan. Parser uses stateful line-by-line processing with pending directive accumulation, body mode transitions, and variable substitution during parsing. Post-processing should preserve `parse_timeout_value()`, `parse_condition()`, `substitute_variables()` helpers while pest grammar handles tokenization and precedence. Request-variable substitution (`{{req.response.body.$...}}`) and function calls (`guid()`, `upper()`) must remain in processor, not parser, as they depend on execution-time context. Round-trip parity with serializer is critical—serializer round-trip test must pass. All 48 existing parser tests define the contract and must pass without modification.
- **2026-03-20 parser-contract baseline landed.** `src/core/src/parser/tests.rs` now freezes blank-line body mode, raw `@...` body lines, directive buffering for the next request, precedence between body content/comments/IntelliJ script blocks/assertions/request lines, and README/reference examples. `src/core/src/serializer.rs` now covers multi-request round-trips so directive boundaries stay stable during the pest rewrite.
